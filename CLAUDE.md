# SOXX Regime Monitor — CLAUDE.md

## Project Overview

Public, zero-cost dashboard that computes and displays the SOXX semiconductor ETF regime state machine (RISK-ON / MONITOR / OUT / ACCUM) every trading day. Static site on GitHub Pages, data refreshed via GitHub Actions cron.

**Public URL:** `https://jgiancristofaro.github.io/soxx-regime-monitor/`
**GitHub repo:** `https://github.com/jgiancristofaro/soxx-regime-monitor`
**Owner:** jgiancristofaro

---

## Key Commands

### Pipeline (Python)

```bash
# Install deps (from repo root)
pip install -r pipeline/requirements.txt

# Run the full compute pipeline (generates data/signals.json and data/history.csv)
python pipeline/compute.py

# Run tests
pytest pipeline/tests/ -v

# Run a single test
pytest pipeline/tests/test_state_machine.py::test_trade_golden_record -v
```

### Frontend (Node/Vite)

```bash
cd site

# Install deps
npm install

# Dev server
npm run dev

# Production build (outputs to site/dist)
npm run build

# Preview production build
npm run preview

# Type check only
npm run typecheck
```

### GitHub

```bash
# Trigger data pipeline manually (generates signals.json without waiting for cron)
gh workflow run daily.yml

# Watch workflow run
gh run watch

# Check Pages deployment
gh api repos/jgiancristofaro/soxx-regime-monitor/pages
```

## Architecture

```
GitHub Actions (cron 22:15 UTC weekdays)
  └── pipeline/compute.py
        ├── sources.py: yfinance → stooq → stockanalysis (fallback chain)
        ├── state_machine.py: pure function DataFrame → signals/bands/trades
        └── writes: data/signals.json, data/history.csv

GitHub Pages (deploys on push to main)
  └── site/ (Vite + TypeScript + Chart.js 4.x)
        └── reads data/signals.json at load time (copied into dist/)
```

No servers. No paid APIs. No secrets beyond the built-in GITHUB_TOKEN.

## Data Files — What to Edit Manually

- **`data/manual.json`** — IV30 and P/C OI are computed automatically every trading day by `pipeline/options.py` from the live SOXX options chain (yfinance, free — see `site/public/methodology.html` §2 for the interpolation method). `manual.json` is now a **fallback only**: if the automated fetch fails (network outage, no usable option quotes), `compute.py` keeps whatever is already in this file. Edit it by hand (in the GitHub web UI, site redeploys) only to force an override or bridge a fetch outage:
  ```json
  { "iv30": 0.46, "iv30_asof": "2026-07-01", "pc_oi": 2.9, "pc_oi_asof": "2026-06-18" }
  ```
- **`data/events.json`** — Add chart annotations. Format:
  ```json
  [{"date": "2026-07-10", "label": "TSMC June revenue", "text": "..."}]
  ```

## AI Agent Preferences (User-Specified)

These are hard requirements for all AI agents working on this project:

### 1. Commit and Push Often
- Commit after every meaningful unit of work (individual file, passing test, feature section)
- Push to remote after every commit
- This protects against crashes and saves work incrementally
- Preferred commit message format: `type(scope): description` (e.g., `feat(pipeline): add state machine`)

### 2. WORKLOG.MD — Append-Only Decision Log
- Every significant decision, step, or non-obvious choice MUST be logged in `WORKLOG.MD`
- Format: numbered entry, dated, with a brief rationale
- **NEVER edit old entries** — only append new ones
- Start every AI session by reading the WORKLOG index to understand prior decisions
- Log format:
  ```
  ## [NNN] YYYY-MM-DD — Title
  Decision/step description. Why it was done this way.
  ```
- Keep an index at the top of WORKLOG.MD that lists entry numbers and titles

### 3. Before Starting Work
- Read `WORKLOG.MD` index to understand prior decisions
- Read `CLAUDE.md` to understand project conventions
- Check `data/signals.json` for current state machine output

### 4. When Changing State Machine Constants
- Update `pipeline/tests/test_state_machine.py` fixtures first
- Run `pytest pipeline/tests/ -v` and confirm all pass
- Document the change and rationale in WORKLOG.MD

### 5. Disclaimer — Non-Negotiable
Every page must include: *"Research and education only. Not investment advice. Signals are in-sample on n=3 episodes. No warranty."*

### 6. Methodology Archive — Archive Before Editing
The live methodology is at `site/public/methodology.html`. Historical versions are stored in `site/public/methodology-archive/` and served as part of the static site.

**Workflow: before making any content changes to `methodology.html`:**
1. Copy `site/public/methodology.html` to `site/public/methodology-archive/YYYY-MM-DD.html` (use the date of the edit)
2. Add the archive banner near the top of the copied file (see `2026-07-06.html` for the banner format)
3. Prepend a new row to the table in `site/public/methodology-archive/index.html` (most-recent first)
4. Commit the archive files: `docs(methodology): archive YYYY-MM-DD snapshot before <description of change>`
5. Then make changes to `site/public/methodology.html`

The archive is browsable via the "Methodology Archive →" link at the bottom of the live methodology page. `docs/METHODOLOGY.md` is superseded and points to the live HTML.

## Operating Manual

**Normal operation:** Nothing to do — check the site after US market close. A `signal-change` GitHub Issue means the state changed.

**Pipeline failed (red X on Actions):** Re-run the job from the Actions tab. The site will show a stale-data banner but won't crash.

**IV/P-C stale despite automation:** Check the `daily.yml` run log for "Options fetch failed" — if the live options-chain fetch keeps failing, edit `data/manual.json` in the GitHub web UI as a manual override. Site redeploys automatically.

**Add a chart annotation:** Append to `data/events.json`, commit — site redeploys.

**Change a threshold:** Edit `pipeline/state_machine.py` constants → update tests → document in WORKLOG.MD.
